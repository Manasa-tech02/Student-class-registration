Admin Access Implementation

Design Decisions





Admin creation: Pre-seeded in database via prisma/seed.ts. Admins cannot sign up -- they can only log in.



Login screen: Shared login screen with a Student / Admin toggle. When "Admin" is selected, backend verifies the user's role is "admin". If a student tries to log in as admin, they get rejected.



Signup screen: Only available for students. No changes needed. Admins are never created through signup.



Routing: RootNavigator checks user.role and renders StudentStack or AdminStack accordingly.

2. Update login to validate role -- [src/services/authService.ts](backend/src/services/authService.ts)

Add an optional expectedRole parameter to loginUser():

export async function loginUser(
  email: string,
  password: string,
  expectedRole?: "admin" | "student"
): Promise<TokenPair> {
  const user = await prisma.user.findUnique({ where: { email }, ... });
  // ... password check ...

  if (expectedRole && user.role !== expectedRole) {
    throw new HttpError(403,
      expectedRole === "admin"
        ? "This account does not have admin access"
        : "Please use the admin login for admin accounts"
    );
  }

  return { token: signAccessToken(user.id, user.role), ... };
}

Update [src/controllers/authController.ts](backend/src/controllers/authController.ts) to pass req.body.role to loginUser.

3. Create requireAdmin middleware -- src/middleware/requireAdmin.ts (new file)

export function requireAdmin(req, _res, next) {
  if (req.user?.role !== "admin") {
    return next(new HttpError(403, "Admin access required"));
  }
  return next();
}
